System Model

Universal Architecture Layers

Most production apps can be understood using this model regardless of language/framework.

flowchart TD U[Client: Browser/Mobile/Desktop] --> FE[Presentation Layer\nUI + Interaction] FE --> API[API Contract Layer\nEndpoints + Schemas + Status Codes] API --> BE[Application Layer\nAuth + Business Logic + Orchestration] BE --> DB[(Data Layer\nSQL/NoSQL)] BE --> EXT[External Systems\nPayments/Email/Storage/Analytics]

Presentation

User experience and interaction logic.

Application

Policy, trust, and processing logic.

Data

Persistent source of truth.

Responsibilities

Frontend vs Backend — Precise Boundaries

Frontend responsibilities

  • Render views and components
  • Collect user input
  • Call APIs and show state transitions
  • Client-side validation for UX

Backend responsibilities

  • Authenticate and authorize users
  • Validate input for security/integrity
  • Execute business rules
  • Persist and query data
  • Integrate with external services

Why this split exists

Clients are untrusted environments. Servers are controlled environments. Trust-sensitive decisions must happen server-side.

Contracts

API Contract — Real example

Endpoint: POST /api/auth/login

Request: { email, password }

Response: { token, user, account }

Data Contract — Real example

Table rule: users.email must be unique

Constraint: subscription_tier ∈ {starter, professional, enterprise}

Type Contract — Real example

Frontend interface: type User = { user_id: string; role: 'admin' | 'manager' | 'viewer' }

If backend returns a different shape, TypeScript warns immediately.

Business Contract — Real example

Rule: only admin can invite/remove team members.

Rule: inactive subscriptions can be blocked with 402.

Strong systems make contracts explicit, versioned, and testable.
Runtime

What Happens from URL Entry to First Render

sequenceDiagram participant User participant Browser participant Edge as CDN/Edge participant FE as Frontend Assets participant API as Backend API User->>Browser: Enter URL Browser->>Edge: DNS + TLS + request Edge->>FE: Fetch HTML/CSS/JS FE-->>Browser: Static assets Browser->>Browser: Execute app bundle Browser->>API: Optional bootstrap call (profile/config) API-->>Browser: JSON response Browser-->>User: First interactive render

Understanding this timeline helps you debug slow loads, auth flashes, and startup failures.

Runtime

What Happens on Any Create/Update Action

sequenceDiagram participant UI as Client UI participant Net as HTTP Layer participant App as Backend App participant MW as Middleware participant Store as Database UI->>Net: Send request (JSON + headers) Net->>App: Route request App->>MW: Auth, authz, validation MW-->>App: Approved/Rejected App->>Store: Read/Write transaction Store-->>App: Result App-->>UI: Status + response body UI-->>UI: Update state/re-render

Why this model scales

Clear control points (middleware, data layer, response contract) make behavior predictable under growth.

Data Architecture

Choosing SQL vs NoSQL (General Decision Framework)

DimensionSQL (PostgreSQL/MySQL/SQLite)NoSQL (MongoDB/Redis/etc)
Data shapeStructured, relationalFlexible or key-value/document-heavy
TransactionsStrong ACID supportVaries by engine
Query styleComplex joins/reporting strongFast access for specific patterns
Best forBusiness apps, finance, reportingHigh-scale events, caching, flexible documents

General rule

If your domain has many relationships and strict consistency, start with SQL unless there is a clear reason not to.

API Styles

REST vs GraphQL vs RPC — Why Pick One?

REST

  • Resource-oriented URLs
  • Simple and widely understood
  • Great default for many teams

GraphQL

  • Client chooses response shape
  • Reduces over/under fetching
  • Higher schema/tooling complexity

RPC/tRPC/gRPC

  • Procedure/function-style APIs
  • Strong typing and performance options
  • Different ecosystem tradeoffs
Security

Authentication, Authorization, Validation, Observability

Authentication

Prove identity (JWT, session, OAuth).

Authorization

Enforce permissions (RBAC/ABAC/policies).

Validation

Reject invalid/malicious input at backend boundary.

Observability

Logs + metrics + tracing for debugging production behavior.

No security model is complete without: least privilege + validated inputs + auditability.

Real Code Communication

How Files Communicate in Actual Frontend Code

flowchart LR A[App.tsx\nRoute + Providers] --> B[Page Component\nex: Properties.tsx] B --> C[usePropertyQueries.ts\nuseQuery/useMutation] C --> D[api.ts\nfetchAPI helper] D --> E[/api HTTP request]

Read flow example

  • Route /properties loads page component
  • Page calls useProperties()
  • Hook calls api.getProperties()
  • API helper sends GET /api/properties with Bearer token

Write flow example

  • Mutation calls api.createProperty()
  • Backend returns success JSON
  • React Query invalidates properties and dashboard-stats
  • UI refetches and re-renders with fresh server truth
Real Code Communication

How Services Communicate in Actual Backend Code

flowchart TD Req[HTTP request /api/properties] --> R[Express route in server.js] R --> M1[authenticate middleware] M1 --> M2[requireActiveSubscription] M2 --> M3[checkPropertyLimit + validate schema] M3 --> BL[Business logic handler] BL --> DB[(SQLite via getDb)] DB --> Res[JSON response]

Concrete example: Create Property

The handler in server.js receives POST /properties, applies middlewares, inserts into properties, updates accounts.current_property_count, writes usage_logs, and returns 201.

protectedRouter.post('/properties', checkPropertyLimit, validate(propertySchema), ...)

What this means architecturally

Each backend service has a role: routing service (Express), policy service (middleware), persistence service (SQLite), and integration service (Stripe/webhooks).

Frameworks in Practice

What Is a Framework? (With Your Real Stack)

Framework/ToolWhat it gives youWhat your code providesWhere seen in project

Express HTTP server, routing, middleware pipeline Business logic and route handlers backend/server.js, routes/

Simple definition

A framework is prebuilt structure + lifecycle. Your app plugs business logic into that structure instead of building all infrastructure from zero.

Debugging Map

Where to Look When Communication Breaks

If auth/route access fails

  • Frontend: AuthContext.tsx and App.tsx protected routes
  • Backend: middleware/auth.js (`authenticate`)

If requests return 401/402

  • Frontend: token injection in api.ts
  • Backend: `requireActiveSubscription` logic

If UI stale after write

  • Check mutation onSuccess invalidations
  • Verify query keys match read hooks

If data looks wrong

  • Inspect SQL in route handlers
  • Confirm account_id filtering exists
  • Check response-field mapping for frontend
